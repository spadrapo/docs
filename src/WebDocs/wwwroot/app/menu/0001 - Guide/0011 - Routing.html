Routing

Drapo includes a client-side SPA routing system that maps URL patterns to Drapo function calls — typically UpdateSector calls that load different views. Routes are configured on the server and sent to the client as part of the Drapo initialisation payload.

How It Works

Your application needs a default index.html in wwwroot that defines all sectors. When a URL is requested, Drapo loads this shell, then executes the action for the matching route to populate the appropriate sector. See the docs site index.html as a reference.

Defining Routes

Routes are created in your startup configuration using CreateRoute(pattern, action). The pattern is a regular expression and the action is a Drapo function string:

options.Config.CreateRoute(
    "^/function/(?<id>\\w+)$",
    "ClearSector(content);UpdateSector(content,~/app/shared/function.html)");

options.Config.CreateRoute(
    "^/doc/(?<id>\\w+)$",
    "ClearSector(content);UpdateSector(content,~/app/shared/doc.html)");

options.Config.CreateRoute(
    "^/$",
    "UpdateSector(content,~/app/menu/home.html)");

Named Capture Groups

Named groups in the regex (e.g. (?<id>\w+)) are extracted and stored in the special __route storage object. They are available as mustache expressions inside the action string — so {{id}} in the action is replaced by the captured URL segment.

Navigating with ApplyRoute

Use the ApplyRoute function to navigate programmatically. It updates the browser's address bar and executes the matching route action without a full page reload:

<a d-on-click="ApplyRoute(/page/about)">About</a>
<button d-on-click="ApplyRoute(/page/{{item.slug}})">Open</button>

Rejected Routes

Some URL paths should not be handled by the router (e.g. API endpoints, static files). Rejected routes match a pattern but stop processing immediately without executing any action:

options.Config.CreateRejectedRoute("^/api/");
options.Config.CreateRejectedRoute("^/admin/.*$");

Order matters: place rejected routes before any normal routes that might match the same path.

Dynamic / Multi-Tenant Routes

For multi-tenant applications, register a RouteEvent delegate. It receives the HttpContext and returns a list of routes specific to the current request — tenant, user role, subdomain, etc. Dynamic routes take precedence over static ones:

options.RouteEvent += async (context) =>
{
    string tenant = context.Request.Headers["X-Tenant"].ToString();
    var routes = new List<DrapoRoute>();
    if (tenant == "acme")
        routes.Add(new DrapoRoute {
            Uri = "^/home$",
            Expression = "UpdateSector(content,~/tenants/acme/home.html)"
        });
    return routes;
};

Common context sources for tenant identification: context.Request.Headers, context.Request.Host.Host (subdomain), context.User.IsInRole(), context.Request.Query.

Disabling the Router

Set options.Config.UseRouter = false to disable the client-side router entirely. This is useful for applications embedded inside other pages or iframes where URL changes are not desired.
