Components

Drapo components are reusable, self-contained UI units composed of an HTML template, an optional CSS stylesheet, and an optional TypeScript file. They are loaded by the Drapo middleware and used as custom HTML elements anywhere in a Drapo application. The element tag name matches the component's folder name, prefixed with d-.

File Structure

Each component lives in its own sub-folder under the configured components directory (typically wwwroot/components/):

wwwroot/
  components/
    mybutton/
      mybutton.html    <!-- required: HTML template -->
      mybutton.ts      <!-- optional: TypeScript logic -->
      mybutton.css     <!-- optional: component styles -->
      mybutton.d.ts    <!-- optional: TypeScript declarations -->

Registering Components

Tell the Drapo middleware where to find the components directory in your startup configuration:

app.UseDrapo(options => {
    options.Config.LoadComponents(
        Path.Combine(env.WebRootPath, "components"),
        "~/components");
});

HTML Template

The component template is a standard HTML file. Use d-sector="@" on the root element to give the component its own isolated data context. Custom attributes passed at the usage site are available inside the template.

<!-- mybutton.html -->
<div d-sector="@" class="my-button" dc-text="" dc-onclick="">
    <button type="button">
        <span d-model="{{_text}}"></span>
    </button>
</div>

Key points:

  • d-sector="@" — Creates a unique isolated data scope for each instance of the component.
  • dc-* attributes — Declare custom inputs that callers pass when using the component. They are processed by the TypeScript logic and then removed.
  • Any Drapo attributes (d-model, d-for, d-if, etc.) work normally inside the template.

Using a Component

Once registered, use the component in any HTML page as a custom element with the d- prefix:

<d-mybutton dc-text="Save" dc-onclick="PostData(user)"></d-mybutton>

<d-mybutton dc-text="Cancel" dc-onclick="Navigate(/home)"></d-mybutton>

Drapo replaces the <d-mybutton> element with the rendered template and copies all attributes from the usage site to the template root element.

TypeScript Logic

Add a TypeScript file with the same name as the component folder. Drapo looks for a constructor function named {componentname}Constructor and a class named with the component name (PascalCase). The constructor is called with the rendered element and the Drapo application instance:

// mybutton.ts
async function mybuttonConstructor(el: HTMLElement, app: DrapoApplication): Promise<MyButton> {
    const instance = new MyButton(el, app);
    await instance.Initialize();
    return instance;
}

class MyButton {
    private _el: HTMLElement;
    private _app: DrapoApplication;

    constructor(el: HTMLElement, app: DrapoApplication) {
        this._el = el;
        this._app = app;
    }

    public async Initialize(): Promise<void> {
        // Read custom attributes
        const text = this._el.getAttribute('dc-text') || 'Button';
        const onclick = this._el.getAttribute('dc-onclick') || '';

        // Remove them after reading
        this._el.removeAttribute('dc-text');
        this._el.removeAttribute('dc-onclick');

        // Push data into the component's sector storage
        await this._app.Storage.SetDataKeyField('_text', null, text);

        // Wire up the click handler on the button element
        if (onclick) {
            const button = this._el.querySelector('button');
            button.setAttribute('d-on-click', onclick);
        }
    }
}

Content Projection

Components can receive and render content placed between their tags. Mark the insertion point in the template with d-content="internal". Drapo moves the caller's inner HTML into that slot:

<!-- panel.html (the component template) -->
<div class="panel">
    <div class="panel-header">Panel</div>
    <div style="display:none" d-content="internal"></div>
    <div class="panel-footer"></div>
</div>
<!-- Usage -->
<d-panel>
    <p>This content is projected into the panel body.</p>
</d-panel>

The style="display:none" on the content element prevents a flash of unstyled content before Drapo processes it.

Real Example: The d-code Component

This documentation site uses a d-code component to render formatted code blocks. Its template uses content projection to capture raw HTML and display it as escaped code:

<!-- code.html -->
<div class="code">
    <div style="display:none" d-content="internal"></div>
    <pre></pre>
</div>
// code.ts — Initialize method
public async Initialize(): Promise<void> {
    const elContent = this.GetElementContent();
    const elCode = this.GetElementCode();
    const content = this._app.Document.GetHTML(elContent);
    const contentEncoded = this._app.Document.GetHTMLEncoded(content);
    await this._app.Document.RemoveElement(elContent, false);
    this._app.Document.SetHTML(elCode, contentEncoded);
    elCode.setAttribute('d-pre', 'true');
}

You can see the full source at github.com/spadrapo/docs.

Component Packs (CSS Bundling)

Multiple components can have their CSS bundled into a single HTTP request for better performance. Configure a pack in startup:

options.Config.CreatePack("uicomponents")
    .AddIncludePath("~/components/*")
    .AddExcludePath("*.ts")
    .AddExcludePath("*.map");

TypeScript Type Definitions

The Drapo NuGet package ships TypeScript declaration files. To enable them, add a build target to your .csproj that copies the linked content files to disk:

<Target Name="CopyLinkedContentFiles" BeforeTargets="Build">
    <Copy SourceFiles="%(Content.Identity)"
          DestinationFiles="%(Content.Link)"
          SkipUnchangedFiles="false"
          OverwriteReadOnlyFiles="true"
          Condition="'%(Content.Link)' != ''" />
</Target>

Then create a tsconfig.json in your project:

{
  "compilerOptions": {
    "target": "ES2017",
    "module": "none",
    "lib": ["ES2017", "DOM"],
    "outDir": "./wwwroot/components/",
    "types": ["drapo"]
  },
  "include": ["wwwroot/components/**/*.ts"],
  "exclude": ["wwwroot/components/**/*.d.ts", "wwwroot/components/**/*.js"]
}

Component Lifecycle

  1. Discovery — Drapo scans the DOM for elements with the d- prefix tag name.
  2. Loading — The component's HTML template is fetched from the middleware (or cache).
  3. Replacement — The custom element is replaced with the rendered template. Attributes from the usage site are transferred to the template root.
  4. Content Injection — If the component has a d-content="internal" slot, the caller's inner HTML is moved there.
  5. Instantiation — The TypeScript constructor function is called, if present.
  6. Data Binding — All d-* attributes in the resolved template are processed normally.

Best Practices

  • Always use d-sector="@" on the template root to keep component data isolated from the page.
  • Use the dc-* attribute prefix for custom component inputs, and remove them in Initialize() after reading.
  • Use style="display:none" on the d-content="internal" element to prevent a flash before content projection.
  • Keep the constructor function lightweight — defer heavy work to Initialize().
  • Provide .d.ts declaration files alongside your components for a better TypeScript development experience.
