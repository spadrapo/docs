Data Binding

Data binding in Drapo is the mechanism that keeps the UI automatically in sync with client-side data storage. It uses a mustache-style syntax ({{storageKey.property}}) to read values and the d-model attribute for two-way binding on form controls.

One-way Binding (Read)

Any text node or attribute value that contains a mustache expression like {{user.name}} is automatically updated whenever the user storage item changes. Drapo scans the DOM for these expressions on initialisation and re-evaluates them when data changes.

<span>Hello, {{user.name}}!</span>

Two-way Binding (d-model)

The d-model attribute creates a two-way link between a form control and a storage property. When the user types in the control the storage value is updated immediately, and when the storage value changes programmatically the control reflects the new value.

<input d-model="{{user.name}}" type="text" />

Expression Evaluation

Drapo evaluates mustache expressions in the context of the current data sector. Expressions can reference nested properties ({{order.address.city}}), array items inside a d-for loop ({{item.price}}), or computed values like {{items.length}}. Simple JavaScript-like arithmetic and comparisons are supported inside expressions used in attributes such as d-if and d-class.

Binding Inside Loops

Inside a d-for loop, the iterator variable is available as a special storage context scoped to each iteration. Bindings inside the loop body automatically resolve relative to the current iteration item.

<ul>
    <li d-for="product in products">
        {{product.name}} — ${{product.price}}
    </li>
</ul>

Binding to HTML Attributes

Mustache expressions can appear in any HTML attribute value, not just text nodes. This makes it possible to dynamically set URLs, CSS classes, ARIA attributes, and more.

<img src="{{user.avatarUrl}}" alt="{{user.name}}" />
<a href="/profile/{{user.id}}">View profile</a>

Change Tracking

Drapo automatically tracks changes to storage items. When a value changes, all DOM nodes bound to that storage key are updated in a single pass. This avoids unnecessary re-renders and keeps the UI consistent with the data at all times.
