Drapo Resolver Skill

The Drapo Resolver is an installable AI-agent skill that makes an LLM project-aware: it indexes a Drapo application's own frontend (wwwroot/app + wwwroot/components) and answers go-to-definition / find-references questions about the symbols that app defined — its d-dataKeys, components, and sectors. The shared Drapo MCP knows the framework (every function/attribute that can exist); this skill knows what your app actually contains, so an agent can confirm a binding key, component tag, or sector name really exists instead of guessing.

How to install

The skill is hosted in the spadrapo/docs repository and works with any agent.

Any agent (GitHub Copilot CLI, Claude Code, Cursor) — the cross-agent installer (GitHub CLI 2.90+):

gh skill install spadrapo/docs drapo-resolver --agent github-copilot
# or --agent claude-code / --agent cursor

Claude Code — also available as a marketplace plugin:

/plugin marketplace add spadrapo/docs
/plugin install drapo-resolver@drapo

Update later with gh skill update or /plugin marketplace update drapo.

What it provides

Once installed, the skill auto-activates when relevant and exposes a PowerShell resolver with these commands (all output JSON):

  • resolve-datakey <key> — where a d-dataKey is declared (file, line, d-dataType, URLs, value), or found:false.
  • resolve-component <tag> — the component folder and its files (accepts d-foo or foo).
  • resolve-sector <name> — where a d-sector is declared.
  • find-references <key> — every declaration and reference site (mustache or d-* attribute).
  • scan-unresolved — <d-x> component tags with no matching component (high-confidence), plus UpdateSector sector targets with no static declaration (candidates).

When to use it

  • Before binding to a data key, using a <d-component>, or targeting a sector you did not just author — confirm it exists.
  • After editing Drapo HTML — run scan-unresolved to catch component tags that point at nothing.
  • Before renaming a data key — use find-references to find every binding site.

It is a generic Drapo skill: it only relies on Drapo conventions, so it works in any Drapo application. It runs locally because it reads the app's own files, which the hosted MCP cannot.
