Architecture

Drapo is a declarative Single Page Application (SPA) framework composed of a client-side JavaScript library and a server-side ASP.NET Core middleware that work together to deliver reactive web applications without requiring custom JavaScript.

Overview

Drapo consists of two main components that communicate over HTTP and optionally over WebSockets via SignalR:

  • Client-side library (drapo.js): A TypeScript-compiled JavaScript bundle loaded by the browser. It intercepts HTML attributes prefixed with d-, manages a client-side data storage, performs DOM manipulation, evaluates expressions, handles events, and communicates with the server.
  • Server-side middleware (ASP.NET Core): An ASP.NET Core middleware (UseDrapo) that serves the client library, handles API requests from the client, manages real-time pipes via SignalR, serves component markup and styles, and resolves dynamic routing.

Request Lifecycle

When a user navigates to a Drapo application, the following happens:

  1. The browser loads a static index.html page that includes drapo.js and defines the root sectors using d-sector.
  2. The client library initialises, reads the HTML attributes, and loads data and views into the sectors via AJAX calls to the server.
  3. As the user interacts with the page, the client library evaluates expressions, updates the DOM, triggers functions, and synchronises data with the server as needed.
  4. Real-time updates are pushed from the server to the client via SignalR pipes when enabled.

Client-Side Storage

The Drapo client maintains an in-memory storage registry keyed by d-dataKey values. Each storage item holds typed data (value, object, array, etc.) that can be two-way bound to form controls and rendered in templates. The storage is the central state management mechanism in Drapo applications.

Server-Side Integration

The AddDrapo and UseDrapo calls in Program.cs / Startup.cs integrate Drapo into the ASP.NET Core pipeline. The middleware serves drapo.js and handles special routes for components, views, and pipes. Your own controllers and API endpoints co-exist alongside the Drapo middleware and are called directly by the client library via d-dataUrlGet / d-dataUrlSet attributes.

Folder Convention

By convention, Drapo applications place shared HTML view files under wwwroot/app/, components under wwwroot/components/, and static assets under wwwroot/img/ or similar. The middleware resolves URLs prefixed with ~/ relative to wwwroot, so paths like ~/app/menu/page.html map to wwwroot/app/menu/page.html.
