Pipes

Pipes are Drapo's real-time communication mechanism. They use SignalR WebSockets to push data updates from the server to connected clients without the client needing to poll. When a pipe fires, Drapo automatically refreshes the bound storage items in the browser.

Enabling Pipes

Pipes must be enabled in the server-side configuration and on each storage item that should receive real-time updates.

// In Startup.cs / Program.cs
app.UseDrapo(options =>
{
    options.Config.UsePipes = true;
});

Then on the storage element, add the d-dataPipes attribute with the pipe key to listen on:

<div d-dataKey="notifications"
     d-dataType="array"
     d-dataUrlGet="~/api/notifications"
     d-dataPipes="notificationPipe"></div>

Sending a Pipe from the Server

Any server-side code that has access to the IDrapoPipe (or DrapoPlumberHub) service can broadcast a pipe message. When the message is received by the client, all storage items registered on that pipe key are refreshed from their d-dataUrlGet endpoint.

// Inject DrapoPlumber and send a pipe
await _drapoPlumber.SendPipe("notificationPipe");

Pipe Hub Endpoint

The SignalR hub used by Drapo is automatically mapped by the middleware. Its URL is controlled by options.Config.PipeHubName (default: DrapoPlumberHub). You do not need to configure this manually unless you want to change the hub path.

Polling as an Alternative

When WebSockets are not available or not desired, the Drapo middleware also supports a polling model. The PollingEvent delegate on DrapoMiddlewareOptions lets you provide a function that returns a hash value. The client periodically calls this endpoint and triggers a data refresh whenever the hash changes.

options.PollingEvent += async (domain, connectionId, key) =>
{
    // Return a value that changes when data has changed
    string hash = await _myService.GetDataVersionHash(key);
    return hash;
};

Combining Pipes and Polling

You can use both pipes and polling in the same application. Pipes provide low-latency push updates for scenarios like notifications and dashboards, while polling can serve as a fallback or for coarser-grained cache invalidation.
