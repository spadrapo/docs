Storage

Drapo Storage is the client-side state management system. Each storage item is identified by a unique key defined with the d-dataKey attribute and holds typed data that can be loaded from the server, initialised inline, or manipulated by Drapo functions.

Defining Storage Items

A storage item is declared on any HTML element using the d-dataKey attribute. Additional attributes configure its type, initial values, and server endpoints.

<!-- A simple value -->
<div d-dataKey="username" d-dataType="value" d-dataProperty="Alice"></div>

<!-- An object with properties -->
<div d-dataKey="user"
     d-dataType="object"
     d-dataProperty-name="Alice"
     d-dataProperty-age="30"></div>

<!-- An array loaded from a server endpoint -->
<div d-dataKey="products"
     d-dataType="array"
     d-dataUrlGet="~/api/products"></div>

Storage Types

Drapo supports the following storage types (configured via d-dataType):

  • value – A single scalar value (string, number, boolean).
  • object – A key-value map of properties. Properties are set with d-dataProperty-{name}.
  • array – An ordered list of items, often loaded from a server endpoint.
  • function – Data populated by a Drapo function call.
  • querystring – Data parsed from the current URL query string.
  • parent – Data inherited from a parent sector's storage context.
  • file – File data used for upload scenarios.
  • mapping – A transformation mapping between two storage items.
  • pointer – A reference to another storage item.
  • query – Data derived from a query over other storage items.
  • switch – Data toggled between a set of predefined values.

Loading Data from the Server

Use d-dataUrlGet to load data when the element is rendered, and d-dataUrlSet to define the URL for posting changes back to the server.

<div d-dataKey="orders"
     d-dataType="array"
     d-dataUrlGet="~/api/orders"
     d-dataUrlSet="~/api/orders/save"></div>

Manipulating Storage with Functions

Storage items can be manipulated using built-in Drapo functions triggered by events. Common operations include adding and removing items, clearing fields, and posting data to the server.

<!-- Add an item to an array -->
<button d-on-click="AddDataItem(products, newProduct)">Add</button>

<!-- Remove an item from an array -->
<button d-on-click="RemoveDataItem(products, product)">Remove</button>

<!-- Post storage data to the server -->
<button d-on-click="PostData(user)">Save</button>

<!-- Clear a storage field -->
<button d-on-click="ClearDataField(user, name)">Clear Name</button>

Storage Scope

Storage items defined inside a sector are scoped to that sector's lifetime. When a sector is cleared or replaced, its local storage items are destroyed. Global storage items can be defined outside any sector (e.g., in the root layout) so that they persist across navigation.

Error Storage

The Drapo middleware can be configured with a global error storage key (StorageErrors in DrapoMiddlewareOptions). When a server-side error occurs, Drapo populates this storage item so the UI can reactively display error information.
