d-dataCacheKeys

The d-dataCacheKeys attribute specifies which components should be used to construct the cache key for a data source. This attribute allows you to control how Drapo generates unique cache entries in browser storage (LocalStorage or IndexedDB).

The attribute accepts a comma-separated list of predefined cache key components. Common valid values include:

  • applicationbuild - Includes the application version in the cache key, ensuring cache invalidation on app updates
  • datakey - Includes the data key name in the cache key
  • culture - Includes the current culture/locale in the cache key for localized data

By default, if d-dataCacheKeys is not specified, Drapo uses its own default cache key strategy. When you specify this attribute, you explicitly control which components are used to build the cache key.


Sample 1 - Using applicationbuild and datakey

This is the recommended configuration for most data sources. Including applicationbuild ensures the cache is invalidated when the application is updated, while datakey ensures different data sources are cached separately.



  • {{name.Key}} - {{name.Value}}

Sample 2 - Including culture for localized data

When working with localized data, you can include the culture component to ensure that cached data is specific to each language or locale.



  • {{node.Key}} - {{node.Value}}

Notes

  • The most common configuration is d-dataCacheKeys="applicationbuild,datakey"
  • Using applicationbuild in the cache key ensures that cached data is automatically invalidated when you deploy a new version of your application
  • Using datakey ensures that each data source has its own cache entry
  • Add culture when your data is localized and should be cached separately for each language
  • This attribute works with browser storage (LocalStorage or IndexedDB), not in-memory caching
