d-sector

You can use d-sector to define isolated data contexts and create modular sections in your application. Sectors provide data scoping, content management, and component isolation capabilities.

Basic Usage

The d-sector attribute creates a named section that can have its own data scope and be dynamically updated:

In main sector: {{message}}
In global scope: {{message}}

Sector Types

There are several types of sector values you can use:

  • Named sectors - Use string names for functional areas: d-sector="content", d-sector="sidebar"
  • Numeric sectors - Use numbers for simple isolation: d-sector="1", d-sector="2"
  • Component sectors - Use d-sector="@" for component isolation
  • Dynamic sectors - Use variables for runtime sector names

Component Isolation

The special value @ creates complete data isolation for components:

Shared: {{shared}}
Private: {{private}}
Outside sector - Shared: {{shared}}
Outside sector - Private: {{private}}

Sectors work with [d-dataAccess](d-dataAccess) to control data visibility between parent and child sectors:

Best Practices

  • Use meaningful names - Choose descriptive sector names like "header", "content", "sidebar"
  • Component isolation - Always use d-sector="@" in reusable components
  • Avoid deep nesting - Excessive sector nesting can make data flow complex
  • Document data flow - Clearly document which data is shared between sectors
  • Use with functions - Combine with [GetSector](~/app/functions/GetSector/description.html), [UpdateSector](~/app/functions/UpdateSector/description.html), [SwitchSector](~/app/functions/SwitchSector/description.html), and [ClearSector](~/app/functions/ClearSector/description.html) for dynamic behavior
